Production Machine Learning: Determining ML Technical Debt

The intended audience for this post is both technical and non-technical stakeholders with the purpose of determining and explaining ML Technical Debt. Understanding of ML Technical Debt prevents various stakeholders, e.g., Project Manager, ML Engineers, Data scientists, Customers, and Investors from being blindsided by the excitement of a current ML product to only find out later that the ML product predictions are useless after a few months. Worse still, the expected final ML product never seems to come to a point of deployment value and the ML technical debt cannot be paid off due to financial and time constraints.

Machine Learning (ML) Technical debt is the debt incurred by the deployment of an ML system without developing the code, infrastructure, tools and processes necessary for efficient iteration coupled with a lack of understanding and foresight of the actual ML product requirements. Provision of continuous and actual value for deployed ML systems is hindered by accumulated ML technical debt as the deployed ML system is unable to react to the changes and fails to achieve the necessary performance consistently.

In this post, I have boiled down to two main questions to ask your inhouse-team, outsourced team or invested companies to determine the potential ML technical debt that is incurred.

Question 1: Can the team iterate quickly?

One of the main requirements for a successful ML System in production is the ability to iterate to fix, improve or retain the ML model performance to a level that is constantly valuable to stakeholders.

To understand the potential technical debt of ML systems caused by the failure to iterate or a time-consuming iteration process, we first have to understand the typical lifecycle of an ML system.

The purpose of showing the simplified ML lifecycle diagram is to emphasize the point that ML systems require constant iteration where different modules can interact effectively with each other. Focusing only on modelling without developing a pipeline is insufficient for production ML. Different scale of operations requires different maturity of an ML pipeline so finding what makes sense for your operation is important.

Typically in proof of concept (PoC) deployment, teams do not consider the fact that ML systems require any future updates or continual learning. PoC deployment could be done on a static dataset with no development done to deal with iterating upon new data which is very common in research. PoC deployment can be very deceiving as it tricks stakeholders into thinking that the product is already near completion but it is far away in reality.

The definition of PoC deployment also includes ML systems that only work for a short period of time as the performance degrades over time to become useless while the team is unable to resolve the matter within the degradation period due to the absence of an efficient pipeline. Thus, this includes some ML systems out there that are actually deployed in production and serving customers but are degrading quickly in real-time and will eventually lose out on their value which could cause huge unexpected losses.

Do not be fooled by the time and money taken for a PoC deployment system, extrapolation based on PoC deployment to actual production deployment is dangerous. We need to be aware of the technical debt incurred during a PoC deployment by the team. What was not considered for the ML system to succeed continuously in actual production?

Q1.1: Common production ML problems

  • Data and concept drift.

Data drift is where the distribution of inputs that we trained on changes. Concept drift is where the relationship between the inputs and outputs changes. Drifts can cause our initial successful deployment performances to degrade over time.

  • Additional Feature Requirement

Stakeholders may request additional features midway through the project or even after deployment. The team could also realize that additional features are required to address some deployment issues.

  • Changes in data processing and modelling

The ML team may run some experiments and realize that changing some of the data processing scripts or a change in model architecture is beneficial.

  • Unforeseen issues

Example: Needing to collect a new category of data to properly tackle the problem to provide actual value.

  • Poor performance in deployment

During actual deployment, the team realises that the performance is unsatisfactory despite showing great performance in the research and development stage.

The amount of work and cost needed to address these production issues should be considered ahead of time!

Q1.2: Addressing production issues ahead of time

To deal with these issues, building an entire production system pipeline that can react to new demands and changes while spotting problems will be incredibly beneficial. Each production issue that was mentioned requires the team to be able to iterate and improve upon the current progress.

Going from PoC deployment to an actual ML production system will require the team to ensure that the solution developed is reproducible, scalable and safe; this is where the imperative engineering work comes in. A lot of ML teams are still trying to figure out what is the best MLOps practices and tools for their deployed ML Systems.

Machine Learning applications are rarely static as the external world that it interacts with are constantly changing. For teams that are not thinking about their pipeline to iterate at all, production-ready ML systems will be a far-fetched goal.

ML teams have to take measures to ease collaboration ahead of time by taking into account codebase modularity, clean code practices, and ease of configuration. By setting up a proper system to collaborate, the team can work more efficiently and the development time will be greatly reduced.

From personal experience, there are many commonly overlooked Production ML problems that incur extra technical debt such as:

  • The absence of a monitoring system. Problems in deployment are detected too late or not at all.
  • A poor data management pipeline that lacks DataOps practices leads to poor and inconsistent data quality
  • Solely tracking code with the model. It will be hard to experiment and collaborate for a data-centric approach. This also leads to poor reproducibility.
  • Increasing system complexity over minor accuracy benefits, e.g., changing to new state-of-the-art models with complicated layers that are hard to optimise for.

Lastly, the most overlooked ML problem with the greatest value in my opinion is the creation of valuable datasets. Most ML teams are poor in data collection and sampling techniques; these data techniques are missing because ML engineering efforts typically prioritise more on modelling than on the data.

For example in the data annotation, ML teams may end up just randomly sampling data for annotation instead of focusing their effort on smart sampling techniques. Random sampling can end up driving up the cost (annotation can be very expensive!) without necessarily improving the model performance. The ML Team may also neglect the effort needed to come out with comprehensive annotation guidelines and establish proper collaboration with labellers.

ML teams have to also look into the data collection method as a top priority in the ML pipeline as data collected directly affects model performance and value contribution. The ML team has to ensure that the distribution of data collected is correlated to incoming data distribution in production so that the ML model can potentially provide actual value during deployment. ML teams should not act solely as data consumers but also be involved in the data collection process and work to bring in new ideas to improve the process.

Concluding Question 1:

Understand that Technical Debt can accumulate in any part of the ML systems. It is wise to think ahead of time regarding potential scenarios on how the problem might change in the future and determine from there how fast can the ML system in production change and improve according to the new demands. This will provide a more comprehensive picture of the effort, time and cost required to build a continuously successful product and not get fooled by early prototyping success.

Question 2: How good must the model perform to be valuable?

We need to define what metrics are important to us and at what performance for all metrics that was defined will the model be deemed valuable to our stakeholders.

The determination of value should be discussed among stakeholders to properly scope the project requirements to match expectations. Different problems require different performance requirements to be of value (remember to include business metrics into the equation too!). The higher the performance requirement, the higher the amount of technical debt that we could be blindsided by.

Problems with a high cost of failure and low technology maturity will also increase potential technical debt. The amount of technical debt incurred is driven by:

  • Baseline Human Performance / Performance where it is deemed valuable
  • Cost of Failure
  • Maturity of Technology

A simplified example of 2 different sectors:

  1. A self-driving company requires that the traffic sign classification model achieve an accuracy of 99.9999% be valuable. If the accuracy is below, it will not meet safety requirements and cannot be used in actual deployment. The cost of failure could potentially be the loss of lives. To train a classification model to a performance of 99%, the ML engineers were able to do it in a few months but the technical debt is actually huge; to go from 99 to 99.9999% accuracy requires tremendous effort.
  2. A manufacturing company requires that the product defection classification model achieve an accuracy of 99% to be valuable. If the accuracy is below, it will be better to hire multiple manual labours to detect the defective products. The cost of failure will be delivering defective products to customers. To train a classification model to a performance of 99%, the ML engineers were able to do it in a few months.

For these two classification model use cases (self-driving versus manufacturing), the maturity of the technology is the same (ML-based vision classification models) but the cost of failure and performance requirement differs greatly. To go from PoC to actual valuable deployment, the self-driving problem can easily blindside stakeholders due to inflated expectations and the failure to realise the necessary effort to go from initial PoC success to actual production success.

The ML team could also severely underestimate the difficulty in moving from 99 to 99.9999% accuracy (PoC to deployable metric requirement). This is an oversimplified example, the actual challenges are way more complicated and cannot be defined by a single model metric. However, the point of this comparison is not to understand the actual baseline requirements and technology stack of self-driving but to point out the importance of proper project scoping ahead of time. If you are really interested in self-driving requirements, this paper “On a Formal Model of Safe and Scalable Self-driving Cars” is worth a read. Quoting from the paper:

“ consider first that the probability of a fatality caused by an accident per one hour of (human) driving is known to be 10^−6. It is reasonable to assume that for society to accept machines to replace humans in the task of driving, the fatality rate should be reduced by three orders of magnitude, namely a probability of 10^−9 per hour”

The comparison between these 2 problems is made to emphasise the need to clearly define the requirements where the performance can be deemed valuable and understand the scope of the problem.

For both of these examples, the problem can be made easier by keeping humans in the loop by going from full self-driving to automated driving with driver in the loop (L5 to L3) or going from full defect detection to defect detection with workers in the loop to deal with uncertain predictions.

Engineering work can also be put into fixing the environment of the manufacturing factory to be constant for our defection detection problem which in itself resolves a ton of Machine Learning challenges. By working to constraint the environment, the data complexity reduces and less technical rework are typically required. This is also true for the problem of self-driving, many self-driving car companies are constrained to certain areas and weather as of today. However, the key is to understand what will be valuable to our end-users and profitable to the company. The project scoping phase is crucial!

Stakeholders need to be aware of the technology maturity and the required performance to be deemed valuable. Without planning and researching this through, you may just end up building or investing in a Proof of Concept ML System with no production value and incur a great amount of technical debt that can never be paid off.

Concluding Question 2:

By properly scoping and understanding the requirements, the potential technical debt of ML products can be gauged more accurately upfront. Be aware of the effort required to go from a PoC metric performance to the necessary deployable metric performance. Be sure to ascertain the deployable metric requirements and discuss them with all stakeholders, especially the development team (e.g. ML engineers, data engineers and data scientists). Maintaining clear expectations with your stakeholders is important.

Conclusion

Ignorance of ML Technical debt can blindside stakeholders regarding the true cost and time necessary to build ML products of actual and continuous value. Thus, it is important for stakeholders including the ML engineers or data scientists to understand the true effort required to build valuable production-ready ML products and not just focus on their niche areas.

The standard for ML pipelines/systems are still evolving with a lot of new companies building software tools to support a robust ML pipeline that can iterate effectively. No particular solution was introduced to “pay off” ML technical debt upfront or keep it low; it was purposely kept open as the purpose of this article is only to determine potential areas where ML technical debt can accumulate and blindside stakeholders.
